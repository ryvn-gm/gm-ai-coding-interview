# Welcome — Signal/Algorithm Engineer Interview

> 🌐 **Language:** English｜[繁體中文版](welcome.md)

Hi! Thanks so much for taking the time to interview with us. We're really glad you're here.

This note is a quick heads-up about what to expect, so you can show up ready and (hopefully) relaxed.

## What this interview looks like

| | |
|---|---|
| Format | Live coding + a short verbal design discussion |
| Duration | About 90 minutes total |
| Language | Python |

We'll work through three short tasks:

1. **Task 1 (~30 min)** — Add a feature to an existing Python codebase
2. **Task 2 (~40 min)** — Analyze a dataset and validate a product claim
3. **Task 3 (~15 min)** — A verbal system design discussion (no code required)

There are a few minutes of Q&A after each task.

## What you'll need

- **Your own laptop**, with a working Python environment
- **A few packages installed**: `numpy`, `scipy`, `pandas`. If you don't have them, just install them before we start — no rush.
- **Whatever editor you prefer** — VS Code, PyCharm, vim, anything goes

## About AI tools

**You can use any AI tools you want. No restrictions, no caveats.**

Seriously — ChatGPT, Claude, Copilot, Cursor, whatever you'd reach for on a normal workday. We don't care *whether* you use AI; we care about *how* you use it and how you reason about what it gives you back.

You can paste an entire file into Claude and ask "what's wrong with this?" That's totally fine. The interesting part is what you do with the answer.

## A few things that'll really help you do your best

These aren't graded directly, but they'll genuinely help us see your strengths. We'd love it if you could keep them in mind throughout.

### 1. Talk us through what you're thinking

We care about *how* you think, not just what you ship. So please — talk out loud as you work. Things like:

- "I'm going to read through this file first to understand the architecture"
- "I'm a bit suspicious of X, my hunch is Y because…"
- "I want to ask the AI about Z — let me write the prompt"
- "Hmm, I'm stuck. Let me try a different angle"

If you go quiet and code in silence for 15-20 minutes, we honestly can't tell what's going on in your head, and that hurts you more than it helps. Even saying "I'm thinking about how to handle the edge case" gives us something to work with. **When in doubt, just say what's on your mind.**

### 2. The questions are intentionally a bit fuzzy

Some parts of these tasks are deliberately ambiguous. Real-world requirements are *always* fuzzy, and we want to see how you navigate that. Here's the move we're hoping for:

1. **Spot the fuzzy bit**: "I'm not sure what's meant by X here"
2. **Make a reasonable assumption**: "I'm going to assume X means Y, because…"
3. **Keep moving with that assumption**: "Based on that, I'll do…"

Please don't get stuck saying "the question isn't clear enough." You're welcome to ask us clarifying questions (we'll answer some), but the better move is to make a reasonable assumption out loud and then go. **We're genuinely fine if your assumption is different from what we had in mind** — what matters is that you made one and explained why.

### 3. Look stuff up freely

Forgot how a numpy function works? Not sure about a stats concept? **Just look it up or ask the AI.** Seriously, this won't count against you.

Guessing blindly looks worse than checking. "I don't know, but I can find out in 10 seconds" is something we actively want to see — it's how good engineers actually work.

## One last thing

We know interviews can feel weird and high-stakes, no matter how friendly the company is. We want to be upfront: **this isn't a trick interview.** No gotchas, no leetcode hards, no "we were really hoping you'd say *that exact word*."

What we're actually trying to figure out is pretty simple: would we enjoy working with you? That's it.

So: relax, be yourself, ask us questions whenever you want, take a break if you need one, and — if you can manage it — try to have a bit of fun with it.

Looking forward to meeting you!
