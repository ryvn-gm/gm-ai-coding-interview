## 職能框架

### 核心設計原則

**Critical Thinking（CT）是貫穿所有 task 的元能力**，不列入職能排序，而是獨立評分。CT 不只是「能不能 step back」，而是整個思考過程：**質疑前提、識別假設、判斷資訊可信度、在模糊情況下找到真正的問題所在。** 每個 task 都刻意埋入一個前提陷阱，測的不是能不能解題，而是有沒有先質疑題目本身。

CT 在每個 task 中都會獨立評分（見 BARS 量表），不與下列職能並列比較。

### 關鍵職能

**Must-have（門檻）— 每項至少達 BARS 3 分方可錄取：**

| 排序 | 職能 |
|------|------|
| 1 | AI 協作判斷力 — 知道何時用 AI、何時質疑 AI、何時不信 AI |
| 2 | 問題分解能力 — 把模糊需求拆成可驗證的最小步驟 |
| 3 | Debug 能力 — 從症狀追根因，不侷限在當前層 |
| 4 | 訊號處理基礎知識 — filter、FFT、peak detection 原理 |

**Important（加分）— 用於區分同分候選人：**

| 排序 | 職能 |
|------|------|
| 5 | 快速學習 / 快速迭代 — 陌生領域快速上手並產出（Task 2 整合測試）|
| 6 | 演算法驗證能力 — 不盲信結果，能設計驗證方式 |
| 7 | Embedded 工程意識 — 記憶體、功耗、定點數的 trade-off |

### AI 審查標準（重要說明）

整段 code 丟給 AI、整段給 AI 審查 ✅ 完全可以。

判斷的關鍵在於：
- 給 AI 的問題是「幫我做 X」還是「這個假設成立嗎」
- 拿到 AI 輸出後，有沒有帶著自己的判斷去評估
- 能不能解釋 AI 說了什麼，而不只是複述

## Task 1 — Overengineered Codebase · 加新功能

### 目標

測試候選人在面對過度設計的 codebase 時，能否在動手前先質疑需求與架構合理性，以及如何運用 AI 工具。

**時間：** 30 分鐘｜**語言：** Python｜**主要評估：** CT · AI 協作判斷力 · 訊號處理基礎知識

### 給候選人的指令

給他 `task1/rep_counter.py`，唸：

> 「教練想知道哪幾下動作太快，請在不破壞現有介面的前提下，加入 `detect_fast_reps()`，回傳每個 rep 是否低於 800ms。」

### 時間軸

| 時間 | 預期階段 | 面試官觀察重點 |
|------|---------|--------------|
| 0–5 min | 讀 code，理解架構 | 有沒有對架構複雜度表示驚訝或質疑？還是直接接受？ |
| 5–10 min | 提問 or 開始實作 | 有沒有問 800ms 的來源？有沒有問「我該在這個架構上加，還是先討論架構？」 |
| 10–20 min | 實作功能 | 用 AI 時問的是「幫我加功能」還是「這段 code 合理嗎」？AI 輸出有沒有被評估？ |
| 20–25 min | 測試 / 驗證 | 有沒有跑測試？有沒有用邊界案例驗證？ |
| 25–30 min | 結束問答 | 反思問題（見追問題庫） |

### 陷阱說明（面試官知道，不告訴候選人）

**前提陷阱 ①：** 「太快」是相對於**固定 800ms**，還是相對於**這位用戶自己的節奏基準**？實作 800ms 固定閾值完全可以，但從來沒質疑過這個假設的人，在真實產品上遲早會出問題。

**前提陷阱 ②：** 現有 codebase 的複雜度（Kalman + adaptive threshold + dataclass 套 dataclass）是本來就不需要的。CT 強的人會在加功能之前先問：「我是要在這個架構上加，還是先討論這個架構合不合理？」

**Codebase 刻意埋的問題：**
1. `gaussian_filter1d` + `KalmanSmoother` 雙重 smoothing — bandpass 之後已夠乾淨，Kalman 多餘
2. `_compute_adaptive_threshold` 是 O(n × window) 純 Python 迴圈 — 乾淨資料不需要 adaptive，10 行固定 threshold 就能解決
3. `hash(raw_signal.tobytes())` 做 cache — 這個 use case 完全沒有 cache 的理由
4. 4 個 dataclass（`FilterConfig`、`KalmanState`、`PeakConfig`、`RepResult`）+ 4 個 class 包裝一個本質上是「找 peaks」的功能
5. `FilterConfig` 和 `PeakConfig` 都有 `smoothing_sigma`，重複定義且來源不明

| 觀察行為 | 對應職能 | BARS 參考 |
|----------|---------|----------|
| 問「800ms 是怎麼來的？是固定標準還是個人基準？」 | CT | 主動質疑 → 4-5 分 |
| 指出 codebase overengineered（即使選擇先完成功能） | CT | 識別假設 → 4-5 分 |
| 沒質疑但結束後被問到能立刻意識到 | CT | 被引導後反應 → 3 分 |
| 給 AI 的問題是「這段 code 的複雜度合理嗎」而非「幫我加功能」 | AI 協作 | 帶假設問 AI → 4-5 分 |
| AI 輸出有被評估是否適合這個 codebase 的風格和規模 | AI 協作 | 交叉驗證 → 4 分 |
| 把 AI 輸出直接貼上，不看不改 | AI 協作 | **Red Flag** → 上限 2 分 |

|------|---------|---------|

| 🟡 可接受 | 讀 code 後直接實作 → 用 AI 輔助但有審查 → 功能完成 → 結束問答時能意識到前提問題 | CT 3, AI 3-4 |

以下是一個滿分級別的回答流程，面試官可以用來對照候選人的表現。**不需要完全一致**，重點是思考路徑和關鍵節點有沒有命中。

---

**第一步：讀 code、質疑架構（0–5 分鐘）**

> 「這個 codebase 有 Kalman + adaptive threshold + 多層 dataclass，做的事情其實就是找 peaks，為什麼需要這麼複雜？」

📌 **CT +5：** 讀完 code 第一反應就質疑架構合理性

> 「gaussian_filter1d 之後又做 Kalman smoothing，這是雙重 smoothing，bandpass 後應該已經夠乾淨了。」

📌 **訊號處理 +4：** 能判斷處理步驟是否必要

> 「我先確認一下 — 我是要在這個架構上加功能，還是可以先討論架構合不合理？」

📌 **CT +5：** 在動手前先釐清 scope

---

**第二步：質疑 800ms（5–10 分鐘）**

> 「800ms 是怎麼來的？是固定標準還是根據用戶自己的節奏？不同用戶的動作速度差很大。」

📌 **CT +5：** 質疑需求本身的前提

> 「如果先用固定 800ms 實作，但留個介面讓之後可以改成 adaptive，這樣可以嗎？」

📌 **問題分解 +4：** 務實的分階段策略，不是全有全無

---

**第三步：用 AI 輔助實作（10–20 分鐘）**

> 問 AI：「這段 code 的架構合理嗎？有沒有過度設計的部分？」

📌 **AI 協作 +4~5：** 帶著假設問 AI，而不是「幫我加一個 detect_fast_reps 功能」

> AI 給的 code 有檢查：是否符合現有風格、有沒有不必要的複雜度。

📌 **AI 協作 +4：** 不是直接貼上，有審查和修改

---

**第四步：測試 + 驗證（20–25 分鐘）**

> 用邊界案例測試：恰好 800ms 的 rep、很快的 rep（200ms）、只有 1 個 rep、沒有 rep。

📌 **演算法驗證 +4：** 設計邊界案例而非只跑 happy path

> 「如果 sampling rate 變了，800ms 對應的 sample 數會不一樣，閾值要跟著調。」

📌 **訊號處理 +5：** 理解參數與物理量的關係

### 追問題庫（附參考答案）

根據候選人表現選擇分支：

**已質疑前提 → 深入追問：**

追問：「你會怎麼設計一個 adaptive threshold，讓不同用戶有不同的『太快』基準？」
- 📌 理想：「收集用戶最近 N 次 session 的 rep 時間，算個人均值和標準差，低於 mean - 1σ 就算太快。冷啟動期先用全域預設值。」→ CT +5, 演算法驗證 +4
- 📌 可接受：「用用戶自己的平均值做基準」→ CT +4
- 📌 不及格：「讓教練自己設定」→ CT +2（迴避問題）

追問：「如果這段 code 是你接手的，你會怎麼重構？優先順序是什麼？」
- 📌 理想：「1. 刪掉 Kalman（bandpass 後不需要雙重 smoothing）。2. adaptive threshold 改固定 threshold。3. 拿掉 cache（沒有重複計算場景）。4. 4 個 dataclass 簡化成 1 個。」→ 訊號處理 +5, 問題分解 +5
- 📌 可接受：「感覺太複雜了，我會先簡化 class 結構」→ 問題分解 +3
- 📌 不及格：「這個架構看起來很完整，不太需要改」→ CT +1

**未質疑前提 → 引導後追問：**

追問：「如果我跟你說，不同用戶的動作節奏差異很大，這個功能你會怎麼改？」
- 📌 理想：（停頓）「啊，所以 800ms 是不對的…應該要用用戶自己的節奏做基準。」→ CT +3（被引導後快速反應）
- 📌 可接受：「那可以讓用戶設定自己的閾值」→ CT +3
- 📌 不及格：「那就加一個參數讓外面傳進來就好」→ CT +2（技術解但沒理解本質）

追問：「你覺得現有 codebase 的複雜度合理嗎？如果不合理，問題出在哪？」
- 📌 理想：「不合理。Kalman 多餘、adaptive threshold 不需要、cache 沒用、dataclass 太多層。核心就是找 peaks，不需要這麼複雜。」→ CT +4, 訊號處理 +4
- 📌 可接受：「有點複雜，但我不確定哪個部分不需要」→ CT +3
- 📌 不及格：「挺合理的啊，每個 class 都有它的用途」→ CT +1

**AI 使用相關：**

追問：「你剛才問 AI 的那個問題，如果換一種方式問，你覺得會得到更好的回答嗎？」
- 📌 理想：「我剛才問的是『幫我加功能』，但如果我先問『這段 code 有沒有過度設計』，AI 可能會幫我先識別問題，而不是在有問題的架構上繼續加東西。」→ AI 協作 +5
- 📌 可接受：「可能可以問得更具體一點」→ AI 協作 +3
- 📌 不及格：「我覺得問得挺好的，AI 給的答案也對」→ AI 協作 +2

追問：「AI 給你的那段 code，你覺得它做了什麼假設？」
- 📌 理想：「它假設 800ms 是對的、假設現有架構是合理的、假設 sampling rate 不會變。」→ AI 協作 +5
- 📌 可接受：「它假設 800ms 是正確的閾值」→ AI 協作 +3
- 📌 不及格：「沒有什麼假設吧，它就是按我說的做」→ AI 協作 +1

## Task 2 — 陌生領域 · 資料有問題的分析任務

### 目標

測試候選人在不熟悉的資料分析領域中，能否先質疑資料品質與前提，而非直接驗證別人的結論。同時觀察快速學習能力和 AI 協作方式。

**時間：** 40 分鐘｜**語言：** Python｜**主要評估：** CT · 快速學習 · AI 協作判斷力 · Debug 能力

### 給候選人的指令

> 「我們有一份戒指用戶的 7 天睡眠 + 每日運動量資料（CSV），PM 說初步分析顯示『運動量越高，當晚睡眠品質越好』，想做成產品功能。請你驗證這個結論，然後告訴我你的發現。開放任何工具，沒有限制。」

| 5–10 min | 發現（或未發現）資料問題 | 有沒有注意到 `sleep_tz` 欄位？有沒有看 steps 分佈？ |

### 陷阱說明（面試官知道，不告訴候選人）

**資料檔案：** `task2/sleep_exercise.csv`

**資料規格：**
- 欄位：`user_id, date, steps, active_minutes, sleep_score, sleep_tz`
- 12 個用戶，7 天（2024-01-08 ~ 2024-01-14），共 84 行
- 直接跑分析的話 r ≈ 0.58，看起來有中等正相關
- 正確對齊時區並移除 artifact 後，相關性接近 0 — **PM 的結論不成立**

**資料問題（三層，越深越難發現）：**

| 層 | 問題 | 影響 | 難度 |
|----|------|-----|------|
| ① | 5 個用戶 `sleep_tz = "UTC"`，其餘 `"UTC+8"`，date 欄位格式一致 | 「運動」和「睡眠」對到錯的一天 | 中 — 需要注意 sleep_tz 欄位 |
| ② | 3 個用戶有多天 `steps > 50,000`（motion artifact，共 11 筆） | 大幅拉高相關性 | 低 — 基本 EDA 就能發現 |
| ③ | 樣本只有 12 人 × 7 天 | 統計上完全不成立 | 高 — 需要統計概念 |

**更深的前提陷阱：** PM 說「初步分析顯示」，這本身就是未經驗證的前提。CT 強的人會先問「這個初步分析是怎麼做的」，而不是直接去驗證一個可能本來就錯的結論。

## Task 3 — 口頭系統設計 · 需求裡藏著錯誤前提

### 目標

測試候選人在口頭設計中，能否釐清模糊需求背後的資料流，識別出對方工程師跳過的前提（rep 偵測發生在戒指/手機端，不能直接變成 HTTP 請求），並設計合理的整合方案。

**時間：** 15 分鐘｜**形式：** 口頭，不開電腦｜**主要評估：** CT · Embedded 工程意識 · 問題分解能力

### 給候選人的指令

> 「我們的戒指 SDK 要整合到第三方健康 app。對方工程師說，他們希望每次偵測到一個 rep，就透過 webhook 即時推送到他們的 server。請你設計這個整合方案。」

### 時間軸

| 時間 | 預期階段 | 面試官觀察重點 |
|------|---------|--------------|
| 0–2 min | 釐清需求 / 直接動手 | 有沒有先問「rep 在哪偵測？誰來發 webhook？」或「資料流是什麼」？ |
| 2–5 min | 提出架構 or 質疑前提 | 有沒有意識到前半段（戒指→手機→server）才是問題？還是直接設計 server-to-server 的部分？ |
| 5 min | **介入點（條件觸發）** | 如果未質疑 → 給提示：「戒指透過 BLE 連手機，這對你的設計有影響嗎？」 |
| 5–8 min | 調整設計 | 收到提示後反應速度？能不能快速轉向？ |
| 8 min | **壓力測試（必問）** | 「如果用戶的手機暫時沒有網路，這個設計怎麼處理？」 |
| 8–12 min | 深入設計 | 替代方案的具體程度（batch sync? pull model? local buffer?） |
| 12–15 min | 結束問答 | 反思問題（見追問題庫） |

### 陷阱說明（面試官知道，不告訴候選人）

**題目故意沒講 webhook 從哪裡發。** 對方工程師說「每次偵測到一個 rep，就透過 webhook 即時推送」，隱含假設是「偵測到 rep」可以直接觸發 HTTP 請求。但他沒想過 rep 是在哪裡偵測到的。

**CT 強的候選人應該問：「rep 是在哪裡偵測到的？誰來發這個 webhook？」**

- 戒指發？→ 戒指沒有網路，不可能
- 手機發？→ 可以，但手機可能沒訊號
- 你的 server 發？→ 可以，但資料要先從戒指到手機到 server

**完整資料流：**

```
戒指偵測到 rep ──BLE──→ 手機 SDK ──網路──→ 你的 Server ──webhook/WS──→ 第三方 Server
                 ↑              ↑                    ↑
              會斷線          可能沒訊號          這段沒問題（server-to-server）
```

**核心問題不是「webhook 技術上不行」，而是對方跳過了整個前半段資料流。** Webhook（或 WebSocket）在 server-to-server 這段完全可行，加上 retry queue 就是業界標準的 reliable webhook。真正的問題在前半段：

| 問題 | 發生在哪裡 | 說明 |
|------|-----------|------|
| 戒指沒有網路 | 戒指 → 手機 | BLE 裝置無法發 HTTP，只能透過 BLE 傳給手機 |
| BLE 會斷線 | 戒指 → 手機 | 手臂角度、距離、干擾都可能中斷 |
| 手機可能離線 | 手機 → Server | 健身房地下室、飛航模式、沒訊號 |
| 推送風暴 | 整條鏈路 | 高強度運動每秒多個 rep，每個都即時推 = 電池耗盡 |

**合理的整合方案（依延遲需求排序）：**

| 方案 | 資料流 | 延遲 | 適合場景 |
|------|--------|------|---------|
| Server relay + WebSocket | 手機 → 你的 Server → WS 推給第三方 | 秒級（取決於前半段） | 第三方需要 near-real-time |
| Server relay + Webhook + retry queue | 手機 → 你的 Server → HTTP POST + 失敗重試 | 秒級～分鐘級 | 標準整合，可靠性高 |
| Batch sync | session 結束後一次上傳，再通知第三方 | 分鐘級 | 大多數分析場景 |
| Pull model | 第三方定期來你的 API 拉資料 | 可控（如每 30 秒） | 第三方有自己的排程 |

**不論哪種方案，手機端都需要 local buffer + 斷線重傳**，這是所有方案的共同前提。

**滿分答案範例：** 「webhook + retry queue 推給第三方沒問題，但前提是手機端要有 local buffer 處理 BLE 斷線和離線場景，而且要問清楚第三方真正需要的是即時性還是數據完整性。」

### 評分原則：方向 vs 名詞

> **這個職位是 Signal/Algorithm Engineer，不是 System Architect。系統設計詞彙（buffer、batch sync、exponential backoff）不是核心要求。**
>
> 評分看的是**思路方向**，不是專有名詞。候選人說「先存起來，有網路再一起送」和說「local buffer + batch sync」，在思路上是同一件事 — 前者給 3-4 分，後者給 4-5 分。名詞只是加分項，不是門檻。
>
> **判斷方法：** 如果候選人方向對但模糊，用追問來區分深度（見追問題庫）。追問後能具體回答「存哪裡、存多少、滿了怎麼辦」→ 4 分。追問後仍然模糊 → 3 分。

**第一步：釐清資料流（0–2 分鐘）**

> 「等等，我先確認一下 — rep 是在哪裡偵測到的？是在戒指上、手機上、還是 server 上？」

📌 **問題分解 +5：** 方案分前後兩段，每段有具體做法
📌 **Embedded +4：** local buffer 設計，考慮斷線重傳

> 「如果第三方只是要做分析，batch sync 就夠了 — session 結束後一次推。如果真的要 near-real-time，那前半段的上傳頻率可以設高一點，比如每 5 秒一批，但要告訴他延遲取決於 BLE 和手機網路，不是我們能控制的。」

📌 **CT +5：** 根據需求彈性調整方案，不是一刀切
📌 **問題分解 +4：** 提供不同場景的方案對照

---

**第四步：回應壓力測試（8 分鐘）**

面試官問：「如果用戶的手機暫時沒有網路，這個設計怎麼處理？」

> 「手機端的 local buffer 就是處理這個的。沒網路時資料存在手機本地，恢復連線後按時間戳排序補傳。buffer 如果快滿了，可以先丟掉最舊的原始資料但保留 summary。要注意的是 buffer 不能無限大，要設個上限，比如 1000 筆或 10MB。」

📌 **Embedded +5：** buffer 有上限、有溢出策略、有優先級
📌 **問題分解 +4：** 考慮到邊界情況（buffer 滿了怎麼辦）

### 預期行為路徑

| 路徑 | 典型行為 | 整體判斷 |
|------|---------|---------|
| 🟢 理想 | 先問「rep 在哪偵測？誰發 webhook？」→ 畫出完整資料流 → 指出前半段是瓶頸 → 提出含 local buffer 的方案（webhook/WS 都行）→ 能區分即時性 vs 完整性 | CT 5, Embedded 4-5 |
| 🟡 可接受 | 開始設計 server-to-server webhook → 被提示後意識到前半段問題 → 補上手機端 buffer → 離線場景有基本解法 | CT 3（上限 4）, Embedded 3 |
| 🔴 不及格 | 直接設計 webhook endpoint → 被提示後仍只討論 server 端 retry → 從未想過資料怎麼從戒指到 server → 沒考慮離線 | CT 1-2, Embedded 1 |

### 追問題庫（附參考答案）

**方向對但模糊 → 用追問區分 3 分和 4 分：**
- 候選人說「先存起來」→ 追問「存在哪裡？手機還是戒指？」（答得出手機 → 4 分）
- 候選人說「不要一起送」→ 追問「那多久送一次？怎麼決定？」（答得出具體策略 → 4 分）
- 候選人說「有上限」→ 追問「滿了怎麼辦？丟掉舊的還是停止收集？」（有取捨邏輯 → 4 分）
- 候選人說「斷線要處理」→ 追問「恢復連線後怎麼知道哪些沒送過？」（有想法 → 4 分）

**已質疑前提 → 深入追問：**

追問：「如果第三方堅持要 near-real-time，你能設計一個折衷方案嗎？延遲 5 秒可接受的話呢？」
- 📌 理想：「前半段（手機→server）設每 5 秒批次上傳，後半段（server→第三方）用 WebSocket 即時推。端到端延遲約 5-10 秒，但要告訴對方：BLE 斷線時延遲會增加，這不是我們能保證的 SLA。」→ CT +5, 問題分解 +5
- 📌 可接受：「把上傳頻率調高就好」→ 問題分解 +3
- 📌 不及格：「5 秒的話用 webhook 就夠了吧」→ CT +2（沒考慮前半段）

追問：「batch sync 的話，session 多長算一個 batch？你怎麼決定這個粒度？」
- 📌 理想：「取決於場景 — 健身 session 結束就是一個 batch（通常 20-60 分鐘）。全天追蹤的話每小時或每次 app 打開時同步。粒度的 trade-off：太細 = 太多 HTTP requests、太粗 = 延遲太高。」→ 問題分解 +5
- 📌 可接受：「一個運動 session 結束就送一次」→ 問題分解 +3
- 📌 不及格：「每天送一次」→ 問題分解 +2（太粗，可能丟資料）

追問：「如果同時有 10 個第三方 app 要整合，你的設計怎麼 scale？」
- 📌 理想：「後半段用 pub/sub 或 fan-out — 資料進 server 後發一次 event，每個第三方各自消費。前半段不變，手機只要上傳到 server 一次。」→ 問題分解 +5
- 📌 可接受：「加一個 dispatch layer，收到資料後分發給所有第三方」→ 問題分解 +4
- 📌 不及格：「每個第三方都從手機直接推」→ Embedded +1（手機受不了）

**被提示後 → 確認理解深度：**

追問：「你現在回頭看，這個需求的根本問題是什麼？是 webhook 技術不行，還是對方跳過了資料流的前半段？」
- 📌 理想：「是對方跳過了前半段。Webhook 在 server-to-server 完全沒問題，問題是他假設 rep 偵測可以直接變成 HTTP 請求，但中間隔了 BLE 和手機這兩層。」→ CT +4
- 📌 可接受：「webhook 可以用，但要先解決資料怎麼到 server」→ CT +3
- 📌 不及格：「webhook 不適合這個場景」→ CT +2（過度簡化）

追問：「如果這個戒指未來加了 Wi-Fi 模組，你的結論會改變嗎？」
- 📌 理想：「有 Wi-Fi 的話戒指可以直接上網，BLE 瓶頸消失。但 Wi-Fi 比 BLE 耗電 100 倍以上，不可能每個 rep 都開 Wi-Fi 傳。所以還是需要 batch，只是可以從戒指直接 batch 到 server，不經過手機。」→ Embedded +5, CT +5
- 📌 可接受：「那戒指可以自己上傳，不用經過手機了」→ Embedded +3
- 📌 不及格：「有 Wi-Fi 就可以用 webhook 了」→ Embedded +2（沒考慮功耗）

追問：「server-to-server 的 webhook 加 retry queue 其實沒問題對吧？那問題到底出在哪？」
- 📌 理想：「對，後半段沒問題。問題出在前半段 — 資料從戒指到 server 這段有 BLE 斷線和手機離線的風險，不是 webhook 本身的問題。」→ CT +4
- 📌 可接受：「對，webhook 可以用，問題是資料來源端」→ CT +3
- 📌 不及格：「webhook 還是有問題吧，不夠即時」→ CT +1（沒抓到重點）

**Embedded 知識追問：**

追問：「BLE 和 Wi-Fi 在功耗上差多少？這對你的設計有什麼影響？」
- 📌 理想：「BLE 一次傳輸大約幾 mA，Wi-Fi 是幾百 mA，差 100 倍以上。即使有 Wi-Fi 也不能一直開著，要用 periodic wake-up。所以即使用 Wi-Fi，batch 還是必要的。」→ Embedded +5
- 📌 可接受：「Wi-Fi 比 BLE 耗電很多，所以不能一直開著」→ Embedded +4
- 📌 不及格：「差不多吧，都是無線通訊」→ Embedded +1

追問：「如果 phone SDK 的 buffer 滿了但還沒連上網，你怎麼處理？」
- 📌 理想：「有幾個策略：1. 丟掉最舊的原始資料但保留 summary（如 rep count + avg duration）。2. 壓縮資料（只保留 timestamp + duration，不保留波形）。3. 通知用戶 storage 快滿了。選哪個取決於什麼資料最重要。」→ Embedded +5, 問題分解 +4
- 📌 可接受：「丟掉最舊的資料」→ Embedded +3
- 📌 不及格：「不會滿吧，手機記憶體很大」→ Embedded +1

### 紅旗

直接開始設計 webhook endpoint 的 auth、retry、rate limit，從頭到尾沒有問過「rep 是在哪裡偵測到的」、「資料怎麼從戒指到 server」，完全跳過前半段資料流。

## 面試官介入規則

### 通用原則

- 候選人卡住超過 **5 分鐘無明顯進展**：可給一次方向提示

- 提示後評分調整：**被提示的職能 BARS 上限降一級**（例如原本可拿 5 分，提示後上限 4 分）
- 壓力測試問題（Task 2 第 20 分鐘、Task 3 第 8 分鐘）**不算提示**，正常評分

## 評估框架

### BARS 評分量表（1-5 分）

每個職能與 CT 元能力各自獨立評分。Must-have 職能每項至少 3 分方可錄取。

#### CT（元能力，獨立評分）

| 分數 | 行為描述 |
|------|---------|
| 1 | 從未質疑任何前提，完成任務即交差 |
| 2 | 遇到問題時在當前層打轉，不主動跳出框架 |
| 3 | 被引導後能質疑前提，能說出一個隱含假設 |
| 4 | 主動質疑需求本身，在 AI 輸出中識別出未驗證的假設 |
| 5 | 主動重構問題定義，能說出「這個結論在什麼情況下不成立」，帶著假設問 AI 而非帶著需求要答案 |

#### AI 協作判斷力

| 分數 | 行為描述 |
|------|---------|
| 1 | 完全不用 AI，或完全依賴 AI 不加審查 |
| 2 | 使用 AI 但僅做表面確認（如「看起來對」） |
| 3 | 能指出 AI 輸出中的一個問題或限制 |
| 4 | 帶著具體假設問 AI，並對輸出做交叉驗證 |
| 5 | 用 AI 作為思考工具，主動設計驗證流程，能清楚解釋為何信/不信 AI 的輸出 |

#### 問題分解能力

| 分數 | 行為描述 |
|------|---------|
| 1 | 無法拆解問題，直接嘗試一步到位 |
| 2 | 能拆成大塊步驟但步驟之間缺乏邏輯連結 |
| 3 | 能將任務拆成 2-3 個可驗證步驟，並依序執行 |
| 4 | 拆解後每步有明確的輸入/輸出預期，卡住時知道卡在哪一層 |
| 5 | 拆解粒度精準，能根據中間結果動態調整步驟，主動定義驗收標準 |

#### Debug 能力

| 分數 | 行為描述 |
|------|---------|
| 1 | 找到問題但說不出為什麼是問題 |
| 2 | 能描述症狀，但只在當前層嘗試修復 |
| 3 | 能從症狀往上追一層找到根因 |
| 4 | 跨層追根因，能解釋因果鏈 |
| 5 | 不僅找到根因，還能指出類似問題可能出現的其他位置，提出系統性修復方向 |

#### 訊號處理基礎知識

| 分數 | 行為描述 |
|------|---------|
| 1 | 說不出任何訊號參數的物理意義 |
| 2 | 知道名詞（filter、FFT）但無法解釋作用 |
| 3 | 能正確解釋 filter / FFT / peak detection 的基本原理與適用場景 |
| 4 | 能判斷特定處理步驟是否必要（如雙重 smoothing 的冗餘） |
| 5 | 能根據資料特性選擇合適的處理方法，並解釋參數選擇的 trade-off |

#### 演算法驗證能力

| 分數 | 行為描述 |
|------|---------|
| 1 | 盲信結果，不做任何驗證 |
| 2 | 做基本 sanity check（如看數值範圍） |
| 3 | 能設計一個測試案例來驗證演算法正確性 |
| 4 | 能設計邊界案例和對照組，主動比對預期 vs 實際 |
| 5 | 主動建立驗證流程，考慮統計顯著性和資料品質對結果的影響 |

#### Embedded 工程意識

| 分數 | 行為描述 |
|------|---------|
| 1 | 完全沒有考慮硬體限制 |
| 2 | 被提醒後能說出記憶體或功耗是考量 |
| 3 | 主動提到 memory / power / latency 中至少一個 trade-off |
| 4 | 能針對設計方案說出具體的資源限制影響 |
| 5 | 能在方案選擇中主動權衡 embedded 限制，提出定點數、buffer 大小等具體考量 |

#### 快速學習 / 快速迭代

| 分數 | 行為描述 |
|------|---------|
| 1 | 面對陌生領域完全無法推進 |
| 2 | 能開始嘗試但方向不明確，耗時過長 |
| 3 | 能快速找到關鍵資訊並產出初步結果 |
| 4 | 初步結果後能快速識別問題並迭代修正 |
| 5 | 在陌生領域快速建立心智模型，迭代方向精準，能將新領域知識與既有經驗連結 |

### 強烈負面信號（Red Flag）

以下行為不自動淘汰，但觸發時**該職能 BARS 評分上限為 2 分**，最終由整體加權分數決定是否錄取。

- **AI 協作：** 任一 task 中把 AI 輸出不加審查直接交差，或 AI 給複雜解全程沒有質疑
- **問題分解：** 無法說明自己的解法分幾個步驟；卡住但不知道卡在哪一層
- **Debug：** 找到問題但說不出為什麼是問題；整段 code 丟給 AI 且自己完全沒有評估
- **知識底線：** 說不出任何訊號參數的物理意義；Task 3 完全找不到 webhook 架構的根本問題

### 整體評估維度

**要的：**
- AI 是工具，他是決策者
- 加功能前先質疑需求本身
- 對「別人給的結論」保持懷疑
- 用 AI debug 但能解釋根因
- 面對陌生領域能快速找到關鍵問題
- 快速迭代，先跑通再優化

**不要的：**
- AI 說什麼就是什麼
- 安靜地完成任務，從不發問
- 直接驗證別人的結論而不先質疑
- 能用 AI 找出問題但說不出為什麼
- 完全不用 AI，全靠自己硬寫
